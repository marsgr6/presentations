§01 · The Motivation

From Applications to Reusable Libraries

Software applications and software libraries develop different engineering skills. While applications emphasize user interaction, libraries emphasize reusable design, abstraction, and computational capabilities.

  • Common practice: Many software capstones focus on dashboards, web applications, or CRUD systems.
  • Educational challenge: A significant portion of the project is often devoted to interface development, authentication, routing, and user experience, leaving less time for algorithm design, data modeling, optimization, and software architecture.
  • Opportunity: Developing a reusable library shifts the emphasis toward API design, modular architecture, testing, documentation, and computational methods— skills that can be reused across many future applications.

Proposed approach: Evaluate students on the quality of their library architecture, public API, documentation, testing, and computational design, rather than on the sophistication of a graphical interface.

§02 · Core Philosophy

Why Libraries Demand Deeper Engineering

Without a visual layout to fall back on, the assessment shifts entirely to the quality of the codebase itself. Students must master core software engineering principles:

  • Managing complexity: The purpose of a library is not simply to implement algorithms—it is to transform complex computational processes into a simple, reusable, and reliable programming interface.
  • API/library Design & Abstraction: Students learn to define clear method signatures, handle variable state, and design interfaces that other developers can adopt intuitively.
  • Modularity & Maintainability: Forcing separation of concerns. The mathematical model must be completely decoupled from data ingestion and output formatting.

§03 · Software Engineering

Managing Complexity Through Software Abstraction

Scientific libraries allow researchers and developers to work at the level of their domain instead of the level of implementation. A well-designed library hides computational complexity behind a simple and reusable programming interface.

  • Simple interface. Users focus on solving problems rather than implementing algorithms.
  • Hidden implementation. Numerical methods, optimization, data structures, parallelism, and hardware acceleration remain inside the library.
  • Reusable software. The same implementation can support many different applications through a stable and documented API.

Key idea: Great libraries reduce complexity by exposing a clean, intuitive API while encapsulating sophisticated computational methods.

§05 · Workflow

Scikit-learn Makes the Estimator Lifecycle Feel Uniform

Scikit-learn reduces a supervised-learning project to a clear, repeatable sequence: import an estimator, define its hyperparameters, fit on training data, and predict on unseen inputs. That same abstraction works across supervised regressors, classifiers, clustering, and many other estimator families.

  • Import: `from sklearn.neural_network import MLPRegressor`
  • Configure: define the model and hyperparameters in one place before training.
  • Train: `model.fit(X_train, y_train)`
  • Infer: `model.predict(X_test)` and evaluate against `y_test`.
  • Abstraction: the same workflow style applies to regressors, classifiers, clustering, and more, so you learn one interface and reuse it everywhere.

§06 · Architecture

Scikit-learn Is Built Around a Single Estimator Contract

Scikit-learn keeps machine learning projects consistent by exposing a small estimator interface and then reusing it across supervised regressors, classifiers, clustering algorithms, preprocessors, and pipelines. That lets you swap models without rewriting the surrounding workflow.

  • Estimator core: models expose a common fit, transform, predict, and score-style API/library.
  • Composition: pipelines and feature unions chain preprocessing and modeling steps together.
  • Shared utilities: model selection, metrics, and validation all plug into the same workflow.
  • Result: one abstraction scales across regressors, classifiers, clustering, and more.
// the API/library stays thin while composition and backends do the heavy lifting

§07 · Professional Development

Beyond Applications: Demonstrating Software Engineering Skills

Developing a reusable library demonstrates competencies that extend beyond implementing a single application. It showcases the ability to design software that can be maintained, extended, and reused by others.

  • Software architecture. Designing modular components, clear abstractions, and well-defined interfaces instead of solving only one specific problem.
  • API design. Creating intuitive, documented, and stable interfaces that allow other developers to build applications on top of the library.
  • Software quality. Developing reusable code supported by testing, documentation, versioning, packaging, and reproducible examples.
  • Professional portfolio. An installable package with documentation and examples provides tangible evidence of software engineering practices and long-term maintainability.

Key takeaway: A library is more than a programming project—it is a reusable software product that demonstrates design, engineering, and collaboration skills.

§08 · Case Study

STAL — A Scientific Library for Spatio-Temporal Analytics

STAL (Spatio-Temporal Analytics Library) illustrates how a capstone project can evolve from a single-purpose application into reusable scientific software.

  • The challenge. Spatio-temporal data analysis often requires repeated workflows for data cleaning, feature engineering, spatial operations, forecasting, and visualization across multiple projects.
  • The solution. STAL encapsulates these workflows into modular, reusable components accessible through a consistent public API.
  • The outcome. Instead of rewriting analysis pipelines for every new dataset, researchers can reuse tested implementations, improving productivity, reproducibility, and software quality.
09 / 15
§09 · Architecture

The STAL Library Is Organized as a Thin Public Surface Over a Deep Stack

STAL is designed so researchers only touch a small, expressive public API/library, while the real work happens in domain modules, scientific packages, and lower-level computational backends. That separation keeps the interface simple without sacrificing performance or flexibility.

  • Researcher-facing API/library: a compact entry point for forecasting, clustering, interpolation, and other tasks.
  • Domain modules: feature engineering, optimization, spatial analysis, and visualization remain modular and reusable.
  • Scientific stack: NumPy, SciPy, PyTorch, TensorFlow, Rasterio, GeoPandas, and Xarray provide the heavy lifting.
  • Backends: Python, Numba, CUDA, C++, and Rust give STAL room to scale without changing the public interface.

§13 · Success Criteria

Redefining the Definition of Done

A library is complete not when it "works once," but when it can be understood, reused, tested, and extended by other developers.

  • Reliable implementation. Core functionality is validated through automated testing, reproducible results, and performance evaluation.
  • Well-designed API. Public functions and classes are intuitive, consistent, documented, and stable, allowing developers to build applications with minimal effort.
  • Reusable software product. The library is packaged for distribution, accompanied by documentation, examples, tutorials, and version control to support long-term use and maintenance.
  • Engineering quality. The codebase follows established software engineering practices, including modular architecture, continuous integration, code review, and maintainability.
Success is measured not only by correct algorithms, but by the quality, reusability, and sustainability of the software delivered.

§14 · Conclusion

Expanding the Capstone Vision

Software applications will always remain an essential capstone option. However, modern software engineering also depends on reusable libraries, frameworks, and developer tools that enable entire ecosystems of applications.

  • Broader learning opportunities. Students can choose projects aligned with their interests, whether building applications, scientific libraries, frameworks, or developer toolkits.
  • Deeper software engineering. Library development emphasizes abstraction, API design, modular architecture, testing, documentation, packaging, and long-term maintainability.
  • A stronger software portfolio. Graduates leave with software products that demonstrate reusable engineering solutions, not only completed applications.
